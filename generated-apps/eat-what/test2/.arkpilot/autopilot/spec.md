# 今天吃什么 - 产品规格说明书

## 产品价值主张

每天最纠结的三件事之一就是"今天吃什么"。本 App 通过"随机转盘 + 食材智能推荐 + 心情/时间筛选"三层决策机制，帮助用户在最短时间内决定一餐吃什么，并把饮食选择变成有趣、可记录、可分享的日常生活体验。

核心承诺：**10 秒内，从"不知道吃什么"到"好，就吃这个"**。

## 产品类别与核心使用场景

- **产品类别**：生活决策 / 饮食助手类移动应用
- **目标平台**：HarmonyOS 普通手机（非折叠、非平板）
- **核心使用场景**：
  1. 餐前纠结场景：用户即将吃饭但不知道吃什么，打开 App 快速决策。
  2. 冰箱剩余食材利用场景：用户想根据现有食材决定做什么菜。
  3. 饮食记录与复盘场景：用户记录每日三餐，形成打卡习惯并查看历史。
  4. 社交分享场景：用户将今日餐食决定生成分享卡片，发送给朋友或社交平台。

## 核心任务（Jobs-to-be-Done）

1. **随机抽菜决策**
   - 通过转盘/抽卡形式，从预设或自定义的菜谱池中随机选出一道菜。
   - 支持重新抽取和锁定结果。

2. **按冰箱食材推荐菜谱**
   - 用户输入或管理冰箱里已有的食材。
   - App 根据食材组合推荐可制作的菜谱。

3. **按心情/时间/场景筛选**
   - 用户可选择当前心情（如开心、疲惫、想吃辣、想清淡等）、可用时间（快手菜/正餐/周末大餐）、餐别（早/午/晚/夜宵）。
   - 筛选结果影响推荐和转盘的候选池。

4. **每日饮食打卡记录**
   - 用户可记录今天实际吃了什么，添加备注、满意度评分。
   - 支持按日历视图查看历史打卡。

5. **生成并分享餐食卡片**
   - 将某次决策结果或打卡记录生成精美的分享卡片。
   - 卡片包含菜名、配图占位、决策理由、日期等信息，支持保存/分享。

## 页面结构

| 页面 ID | 页面名称 | 说明 |
|---------|----------|------|
| `home` | 今日决策首页 | 核心入口，展示推荐菜谱、转盘快捷入口、今日打卡状态。 |
| `wheel` | 随机转盘 | 全屏转盘交互，抽取今日菜品。 |
| `ingredients` | 冰箱食材 | 管理食材清单，查看基于食材的推荐。 |
| `filter` | 筛选条件 | 选择心情、时间、餐别等筛选维度。 |
| `records` | 饮食打卡 | 日历/列表形式展示打卡记录，支持新增。 |
| `share-card` | 分享卡片 | 生成并预览分享卡片。 |

## 导航模型

- **主导航**：底部 Tab Bar，固定在屏幕底部。
  - Tab 1：今日（`home`）
  - Tab 2：食材（`ingredients`）
  - Tab 3：打卡（`records`）
- **次级导航**：
  - 从 `home` 可进入 `wheel`、`filter`。
  - 从 `ingredients` 可进入 `filter`。
  - 从 `records` 可进入打卡详情/编辑页，以及 `share-card`。
  - 任何决策结果页均可进入 `share-card`。

## 数据源策略

- **菜谱数据**：内置一份轻量菜谱库（菜名、标签、所需食材、预计用时、适合餐别、难度）。菜谱库随应用包发布，可本地读取。
- **食材数据**：用户自行维护冰箱食材清单，持久化存储在本地 Preferences 或 App Storage。
- **打卡记录**：本地持久化存储，包含日期、餐别、菜名、满意度、备注。
- **筛选条件**：运行时状态，不持久化或仅记住最后一次选择。
- **默认状态**：首次打开应用时不强制预置示例数据，所有用户数据从空状态开始；视觉设计稿中的示例数据仅用于展示效果。

## 组件结构

- **底部导航栏（BottomTabBar）**：3 个主 Tab，固定底部。
- **转盘组件（DishWheel）**：可旋转的扇形转盘，带高亮指针和结果弹窗。
- **食材卡片（IngredientCard）**：展示食材名称、数量、过期提示，支持删除。
- **菜谱卡片（DishCard）**：展示菜名、标签、用时、匹配度，点击进入详情。
- **筛选面板（FilterPanel）**：标签/胶囊选择器组合。
- **打卡表单（RecordForm）**：日期选择、餐别选择、满意度评分、备注输入。
- **日历视图（RecordCalendar）**：月份切换、日期打卡标记。
- **分享卡片（ShareCard）**：固定宽高比，含菜品信息、日期、装饰元素，支持导出为图片。
- **空状态组件（EmptyState）**：用于无食材、无记录、无推荐结果等场景。

## 非目标

- 不实现联网搜索菜谱。
- 不实现用户账号登录与云端同步。
- 不实现 AI 对话式推荐。
- 不实现食材购买/电商链接。

## 验收标准

1. 用户可在首页一键进入转盘并完成随机抽菜。
2. 用户可管理食材并查看基于食材的推荐。
3. 用户可设置心情/时间/餐别筛选，筛选结果影响推荐与转盘候选池。
4. 用户可记录每日饮食打卡并查看历史。
5. 用户可将决策结果或打卡记录生成分享卡片。
6. 应用在目标设备分辨率下正常显示，底部 Tab 导航可正常切换。

## Target Device

- 屏幕分辨率：1272vp × 2756vp（宽 × 高）
- 设备类型：普通手机（非折叠机、非平板）
- 像素密度倍率：3x（即 HTML 设计稿 390px = 设备上 1170vp 物理像素，但 ArkTS 使用 vp 单位时 1px≈1vp）
- 布局策略：内容区 max-width 约束在 390-420vp 内居中，禁止全宽铺满。**注意：Tab 导航栏等必须在底部，且若使用 `position: fixed` 必须限制宽度与 body 一致（如 `width: 100%; max-width: 390px; margin: 0 auto; left: 0; right: 0;`），严禁在 PC 浏览器上全宽拉伸。严禁生成在侧边（如右边/左边）**。
- 特殊注意：2756px 高度意味着屏幕很长，滚动区域充足，但需确保首屏关键内容（Header + 核心卡片）在顶部 600vp 内可见
