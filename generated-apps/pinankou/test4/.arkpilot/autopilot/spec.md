# 平安扣 Pinankou — 产品规格说明书

## 1. 产品概述

### 1.1 Value Proposition

「平安扣」是一款面向企业/园区/社区安全管理员的移动端工具，用极简的本地化优先方式把安全事项记录、风险隐患上报、巡检打卡、应急联系、整改进度跟踪和统计看板整合到一部手机中。它让一线人员无需复杂培训即可「随手记、随时报、随时查」，把安全责任从纸面落实到指尖，降低漏报与整改滞后带来的风险。

### 1.2 Product Category

企业/园区/社区安全管理工具（B2E / 基层治理 SaaS 移动客户端）。

### 1.3 Primary User Scenario

王师傅是某科技园区物业安全员。每天早上 8:30，他打开「平安扣」查看今日待巡检点位，到达每个点位后拍照打卡并记录设施状态；巡检中如果发现消防通道堆物、地面破损等隐患，立即拍照、选择风险等级并上报；中午在首页查看「待整改」隐患的负责人和进度；遇到突发情况，一键呼叫应急联系人；每周五通过统计看板向主管汇报本周隐患发现量、整改完成率和平均整改耗时。

## 2. 核心 Jobs-to-be-Done

1. **记录安全事项**：快速创建日常安全相关事项（如设备检查、培训、会议、整改督办），附加照片、位置、负责人和备注。
2. **上报风险隐患**：发现隐患后选择类型与风险等级（低/中/高/重大），拍照取证，指派整改人并设置期限。
3. **巡检打卡**：按预设巡检路线逐点打卡，支持扫码/定位/手动确认，记录异常并关联隐患上报。
4. **管理应急联系人**：维护常用应急联系人（消防、安保、医疗、主管等），支持一键拨号。
5. **跟踪整改进度**：查看隐患/事项的整改状态（待处理/处理中/待验收/已完成/已关闭），添加进度更新与验收记录。
6. **查看统计看板**：按日/周/月查看隐患趋势、整改完成率、巡检完成率、风险等级分布等关键指标。

## 3. 页面结构

| Page ID | 页面名称 | 说明 |
|---------|---------|------|
| `home` | 首页 / 工作台 | 今日概览、待办提醒、快捷入口、最近动态 |
| `patrol` | 巡检打卡 | 巡检路线与点位列表、打卡、异常上报入口 |
| `issues` | 隐患/事项 | 隐患与安全事项列表、筛选、搜索、新建入口 |
| `contacts` | 应急联系人 | 联系人列表、分组、一键拨号、新增/编辑 |
| `stats` | 统计看板 | 图表与指标、时间维度切换 |

### 3.1 弹窗/表单/次级页面

- `issue-form`：新建/编辑隐患或安全事项。
- `issue-detail`：隐患/事项详情与整改进度时间线。
- `patrol-checkin`：巡检点位打卡确认。
- `contact-form`：新增/编辑应急联系人。

## 4. 导航模型

底部 Tab 导航（5 个主 Tab）：

1. **首页** `tab-home`
2. **巡检** `tab-patrol`
3. **隐患** `tab-issues`
4. **联系人** `tab-contacts`
5. **统计** `tab-stats`

次级页面通过路由跳转：`pages/IssueForm`、`pages/IssueDetail`、`pages/PatrolCheckin`、`pages/ContactForm`。

## 5. 数据策略

- 采用本地优先（Local-first）存储：首屏数据默认从 `AppStorage` / `PersistentStorage` 读取，无网络也可使用。
- 数据模型保持扁平，主要实体为：
  - `Issue`（隐患/事项）：id、title、type、riskLevel、status、reporterId、assignee、location、photos、createdAt、dueDate、updatedAt、comments[]
  - `PatrolPoint`（巡检点）：id、name、area、sequence、status、lastCheckAt、lat、lng
  - `PatrolRecord`（巡检记录）：id、pointId、checkedAt、photos、notes、abnormal、relatedIssueId
  - `Contact`（应急联系人）：id、name、role、group、phone、priority
  - `StatsSnapshot`（统计快照）：period、issueCount、resolvedCount、patrolCompletionRate、riskDistribution、avgResolutionHours
- 不强制要求服务端；若未来对接 API，统一通过 `SafetyService` 抽象层切换。
- **首运行状态**：各列表默认空，显式展示空状态插图与引导文案。示例数据仅出现在 HTML 视觉稿中，不作为产品默认种子数据。

## 6. 非目标

- 不实现实时音视频通话或即时消息聊天。
- 不实现复杂权限矩阵与多级审批工作流。
- 不实现 AI 图像识别与自动定级。
- 不实现离线地图导航；仅记录点位与打卡状态。

## 7. 验收标准

1. 用户可在 3 步内完成隐患上报（选择类型 → 拍照/备注 → 提交）。
2. 巡检列表清晰展示已完成/待巡检/异常点位，打卡后状态即时反馈。
3. 隐患列表支持按状态与风险等级筛选，并能进入详情更新进度。
4. 应急联系人支持分组展示与一键拨号。
5. 统计看板至少展示 4 个核心指标与 2 个趋势图。
6. 应用构建通过 ArkTS 严格模式（strict mode）检查。
7. 目标设备下首屏关键内容（Header + 核心卡片）在 600vp 内可见。

## 8. 未决问题

1. 是否需要接入园区真实巡检路线数据，还是本地预置几条示例路线？（默认本地预置一条示例路线用于体验）
2. 统计周期默认「本周」还是「本月」？（默认本周）
3. 是否允许用户自定义风险等级名称？（首期固定四级）

## 9. Target Device

- 屏幕分辨率：1272vp × 2756vp（宽 × 高）
- 设备类型：普通手机（非折叠机、非平板）
- 像素密度倍率：3x（即 HTML 设计稿 390px = 设备上 1170vp 物理像素，但 ArkTS 使用 vp 单位时 1px≈1vp）
- 布局策略：内容区 max-width 约束在 390-420vp 内居中，禁止全宽铺满。**注意：Tab 导航栏等必须在底部，且若使用 `position: fixed` 必须限制宽度与 body 一致（如 `width: 100%; max-width: 390px; margin: 0 auto; left: 0; right: 0;`），严禁在 PC 浏览器上全宽拉伸。严禁生成在侧边（如右边/左边）**。
- 特殊注意：2756px 高度意味着屏幕很长，滚动区域充足，但需确保首屏关键内容（Header + 核心卡片）在顶部 600vp 内可见
