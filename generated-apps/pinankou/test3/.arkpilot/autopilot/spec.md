# 平安扣 Pinankou — 产品规格说明书

## 产品定位

**Pinankou（平安扣）** 是一款面向企业班组、物业/园区安全员及一线巡检人员的 HarmonyOS 移动端安全管理工具。它把“安全事项记录、风险隐患上报、巡检打卡、应急联系人、整改进度跟踪、统计看板”六大高频工作聚合到一个 App 内，让安全员在走动式检查中随时记录、即时上报、持续跟踪，降低安全管理的信息断点。

## 产品类别与核心使用场景

- **产品类型**：工具型 B2E（企业赋能）应用 / 也可用于社区/物业安全管理。
- **主要用户**：一线安全员、班组长、物业巡查人员、小微企业 EHS 负责人。
- **典型场景**：
  1. 巡检时发现问题 → 拍照/语音文字描述 → 选择风险等级 → 立即上报隐患。
  2. 班前会上记录今日安全注意事项 → 关联责任人 → 形成待办。
  3. 接收整改任务 → 按节点更新进度 → 拍照留痕 → 闭环验收。
  4. 紧急情况下一键呼叫应急联系人 → 自动带出位置/部门信息。
  5. 月末打开统计看板 → 查看隐患趋势、整改率、巡检完成率。

## 核心价值主张

让一线安全工作“看得见、跟得住、说得清”。通过轻量化记录与透明化跟踪，把安全责任从纸面落到指尖，减少漏检、漏报、漏改，提升团队安全执行力。

## 核心任务（Jobs-to-be-Done）

1. **随手记录安全事项**：快速新增安全事项，支持文字、图片、语音；可关联责任人与截止日期。
2. **隐患上报与分级**：填写隐患描述、位置、风险等级、现场照片，提交后进入整改流程。
3. **巡检打卡留痕**：按预设巡检点逐项确认/异常上报；支持定位与时间戳，生成巡检记录。
4. **应急联系人快速触达**：维护应急联系人名单，支持一键拨号/短信，紧急情况优先显示。
5. **整改进度跟踪**：从上报→指派→整改中→验收→关闭的全链路状态跟踪，支持评论与照片附件。
6. **统计看板概览**：以图表和关键指标展示隐患数量、整改率、巡检完成率、逾期风险。

## 导航模型

- **底部 Tab 导航**（5 个一级入口）：
  1. **工作台**（home）— 入口首页，核心快捷操作与概览。
  2. **巡检**（inspections）— 巡检任务与打卡。
  3. **隐患**（hazards）— 隐患上报与跟踪。
  4. **统计**（stats）— 数据看板。
  5. **我的**（profile）— 个人中心、设置、关于。
- **二级页面**：安全事项、应急联系人从工作台快捷入口进入；巡检详情、隐患详情、联系人详情等通过列表点击 Push 进入。
- **表单/详情页**：以整页表单方式呈现，顶部返回按钮 + 标题 + 保存/提交主按钮。

## 数据源策略

- 本阶段为单端 HarmonyOS 应用，采用**本地持久化优先**策略：
  - 使用 `@ohos.data.relationalStore` / 关系型数据库（RDB）持久化所有业务数据：安全事项、隐患、巡检记录、联系人、统计指标缓存。
  - 所有列表/详情默认读取本地数据库；首次安装启动为**空状态**，不内置示例数据。
  - 图片附件使用应用私有文件目录本地存储，记录图片路径。
- 后续可扩展：
  - 通过 HTTP 接口与企业安全管理平台同步。
  - 增加用户账号体系与多端数据同步。
- **离线可用**：所有核心功能无需网络即可使用，网络恢复后可按需同步。

## 非目标

- 不实现账号注册/登录、权限管理、组织架构管理。
- 不实现复杂的审批流与工作流引擎，仅支持线性整改状态流转。
- 不实现 AI 图像识别或语音转文字（可预留入口，但不作为 MVP 功能）。
- 不实现打印/导出 PDF 报告（可后续扩展）。

## 验收标准

1. 用户可在工作台通过快捷入口新增安全事项、上报隐患、开始巡检、查看联系人。
2. 隐患支持“低/中/高”风险等级与“待处理/整改中/待验收/已关闭”状态流转。
3. 巡检任务支持逐项打卡，异常可一键转为隐患上报。
4. 应急联系人支持一键拨号与短信，列表支持搜索。
5. 统计看板至少展示：隐患总数、整改率、巡检完成率、近 7 日隐患趋势。
6. 所有页面首屏关键内容在 600vp 内可见，底部 Tab 导航不遮挡内容。
7. 应用在目标设备分辨率下完成构建，无阻塞编译错误，静态检查通过。

## 开放问题

1. 是否需要本地生物识别/锁屏密码保护敏感数据？（当前版本不实现，可在设置中预留入口）
2. 巡检任务是否需要支持自定义巡检点，还是由后台/管理员预设？（当前版本支持用户自行创建简单巡检任务）
3. 隐患上报是否需要定位字段？（当前版本支持手动输入位置，定位能力预留）

## Target Device

- 屏幕分辨率：1272vp × 2756vp（宽 × 高）
- 设备类型：普通手机（非折叠机、非平板）
- 像素密度倍率：3x（即 HTML 设计稿 390px = 设备上 1170vp 物理像素，但 ArkTS 使用 vp 单位时 1px ≈ 1vp）
- 布局策略：内容区 max-width 约束在 390-420vp 内居中，禁止全宽铺满。**注意：Tab 导航栏等必须在底部，且若使用 `position: fixed` 必须限制宽度与 body 一致（如 `width: 100%; max-width: 390px; margin: 0 auto; left: 0; right: 0;`），严禁在 PC 浏览器上全宽拉伸。严禁生成在侧边（如右边/左边）。**
- 特殊注意：2756px 高度意味着屏幕很长，滚动区域充足，但需确保首屏关键内容（Header + 核心卡片）在顶部 600vp 内可见
